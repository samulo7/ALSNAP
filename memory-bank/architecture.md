# 云知通仓库文档索引

## 1. 文档目的

本文件用于记录仓库中关键文件、目录和核心文档的作用，帮助 AI 代理和开发者快速理解当前上下文。

## 2. 当前目录说明

### 仓库根目录

当前仓库仍以规划与上下文文档为主，尚未创建前后端代码目录。根目录当前最关键的文件包括 `AGENTS.md`、`system-architecture.md`、`database-design.md`、`module-list.md` 和 `memory-bank/`。

### `memory-bank/`

用于存放项目长期上下文、实施约束和过程记录，是 AI 代理开始编码前必须优先阅读的目录。

## 3. 当前核心文档说明

### `AGENTS.md`

仓库级执行规范，定义文档组织方式、默认技术选型、AI 代理工作顺序与约束。后续任何编码任务都需要先遵守这里的 `Always` 规则，再进入实现。

### `system-architecture.md`

阶段 `A-02` 产出的系统架构基线文档，用于定义阶段 A 的系统边界、模块拆分、页面范围、接口域和核心实体命名。后续工程初始化与接口设计都需要以本文件为上位约束。

### `database-design.md`

阶段 `A-02` 产出的数据库设计基线文档，用于定义阶段 A 的最小数据实体、关键字段和实体关系。后续进入 `A-05` 和 `A-06` 时，应直接以这里的实体命名与关系为准继续细化。

### `module-list.md`

阶段 `A-02` 产出的模块清单文档，用于把 MVP 范围映射到具体模块、页面、接口和数据实体，并明确哪些能力属于后续阶段，防止基础功能范围膨胀。

### `memory-bank/product-design.md`

项目产品设计文档，定义产品目标、目标用户、功能范围、区域化内容方向和 MVP 边界。

当前还额外承担以下实现约束来源：

- 明确学习流程的最小闭环
- 明确测验流程的最小闭环
- 明确销售、技术、新人、内容运营、管理员、访客/演示账号的角色路径
- 明确 MVP 阶段不启用积分体系，只保留轻量通关状态

### `memory-bank/implementation-plan.md`

实施计划文档，定义阶段 A 和阶段 B 的建设顺序、每一步的目标、约束和验收测试。后续实现必须严格按步骤推进，并在每一步通过验证后再进入下一步。

### `memory-bank/tech-stack.md`

技术栈建议文档，定义推荐的前后端框架、数据库、中间件、AI 接入方式和部署监控方向。当前默认基线是 `Next.js App Router + NestJS + Prisma + PostgreSQL + Redis + pgvector + OSS + 阿里云百炼`。

### `memory-bank/progress.md`

进度记录文档，用于记录已经完成的步骤、阶段结果、关键决策、已知边界和下一步计划。后续每完成一个里程碑，都应先更新这里，再继续推进。

### `memory-bank/architecture.md`

仓库文档索引，用于记录当前文件和目录的作用；当后续新增代码目录或设计文档时，需要持续更新。

## 5. 当前架构洞察

- 当前仓库的“架构”仍以文档约束为主，而不是代码结构；实现顺序必须先固化上下文，再创建系统设计，再落到工程代码
- `product-design.md` 已经不仅是愿景文档，也是阶段 A 的行为边界文档；后续页面、权限、测验和学习进度设计都应直接受其第 12 节约束
- `system-architecture.md`、`database-design.md` 和 `module-list.md` 已经把阶段 A 的模块、页面、接口和实体命名统一下来；后续工程实现不应随意再造第二套命名
- MVP 的学习系统是轻量闭环，不是积分运营平台；后续实现时要避免把积分、排行或复杂激励提前混入阶段 A
- 访客/演示账号是独立角色边界，不是管理员降权版；后续在鉴权、导航和后台权限设计中应单独考虑
- 内容平台与测验平台从一开始就需要共享角色与进度语义，后续在 `module-list.md` 和 `database-design.md` 中应保持同一套命名

## 6. 维护规则

- 新增关键目录或文档时，同步更新本文件
- 如果某个文档用途发生变化，先更新本文件再继续实现
- 本文件只记录“文件/目录做什么”，不承担详细系统设计说明

# Repository Guidelines

## 项目结构与模块组织

当前仓库以产品与技术方案文档为主，尚未包含实际业务代码。根目录下的核心文件如下：

- `云知通-阿里云认知加速平台设计文档.md`：产品设计、目标用户与功能范围
- `tech-stack.md`：推荐技术栈、架构分层与落地建议

新增文档默认放在仓库根目录；只有当文档数量明显增多时，再统一拆分到 `docs/`。如果后续开始落地代码，请在同一提交中补齐明确目录，例如 `src/`、`tests/`、`scripts/`。

## 构建、测试与开发命令

仓库当前没有已提交的构建脚本、测试框架或 CI 配置，日常修改以本地校验为主：

- `rg --files`：快速查看仓库文件清单
- `Get-Content .\tech-stack.md -Encoding UTF8`：检查 Markdown 内容与编码
- `rg -n "^(#|##|###)" *.md`：检查标题层级和章节结构

如果后续引入 Markdown lint、文档生成或自动校验工具，请在同一个 PR 中补充本节命令说明。

## 编码风格与命名规范

文档统一使用 UTF-8 编码，正文以中文为主；只有在面向外部协作或术语必须保留英文时，才使用双语或英文补充。Markdown 标题使用标准 `#` 层级，段落简短，列表保持扁平，命令、路径、产品名使用反引号标记。

文件命名应直接体现主题，优先采用简洁、可检索的名称，例如 `system-architecture.md`、`database-design.md`，或与现有风格一致的中文标题。

## AI / Agent 执行规则

以下规则必须被 AI 编码代理优先读取，其中 `Always` 表示始终生效、不得跳过：

- `Always`：写任何代码前，必须完整阅读 `memory-bank/architecture.md`，其中记录仓库关键文件与文档的作用。
- `Always`：写任何代码前，必须完整阅读 `memory-bank/product-design.md`，确保实现与产品目标、学习流程、测验流程和角色路径一致。
- `Always`：每完成一个重大功能或里程碑后，必须更新 `memory-bank/architecture.md`，同步新增文件、目录和关键文档的作用说明。
- `Always`：如果上述 `memory-bank` 文件缺失、不可读或内容明显过期，不得直接开始编码，必须先补齐上下文或明确标记阻塞。

以下规则不是 `Always`，但默认应遵循，用于保证后续代码质量与架构一致性：

- 前端默认采用 `Next.js App Router` + `TypeScript`，优先使用服务端获取数据，避免无必要的客户端请求链。
- 后端默认采用 `NestJS` 模块化设计，接口必须有清晰的 DTO、校验和错误边界，避免把业务逻辑散落在控制器中。
- 数据访问默认通过 `Prisma`，涉及表结构调整时必须同时维护迁移与文档，不允许跳过 schema 同步。
- 网络层应统一封装请求、超时、鉴权和错误处理；禁止在多个页面或组件中复制粘贴请求逻辑。
- 状态管理保持最小化：服务端数据优先交给服务端渲染或请求缓存处理，局部 UI 状态留在组件内，只有跨页面共享状态才引入全局状态管理。
- 涉及 AI、检索、向量或缓存能力时，优先复用既定技术栈：`PostgreSQL`、`Redis`、`pgvector`、阿里云百炼与 `OSS`，不要随意引入平替组件。

## 测试与校验要求

当前没有自动化测试，提交前至少完成以下人工检查：

- 标题层级是否连续、清晰
- 文件名、相对引用、术语是否准确
- 与现有设计文档中的产品口径是否一致

现阶段没有覆盖率指标，质量标准以内容准确、结构清楚、便于评审为准。

## 提交与 Pull Request 规范

当前工作区没有 `.git` 历史，无法从现有提交记录中提炼仓库专属规范。默认使用简短、祈使句式的提交信息，建议带作用域前缀，例如：`docs: 完善技术栈说明`。

PR 应至少包含变更摘要、涉及文件、评审关注点；如果调整了文档结构、术语口径或阅读顺序，应在描述中明确说明。只有当渲染效果本身是评审重点时，才附页面截图。
